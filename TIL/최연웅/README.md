📅 [2/25] AI 모델 튜닝: vLLM 서빙과 GPU 메모리 최적화의 늪
1. 오늘 한 일

AI 과제 수행: SmolLM2-360M-Instruct 모델 vLLM으로 로딩 및 추론 테스트

Colab 환경 (Tesla T4 GPU) 메모리 최적화 트러블슈팅

2. 겪었던 문제 & 해결 과정 (Troubleshooting)

문제 1: T4 GPU의 bfloat16 미지원 에러
내 노트북 내장 그래픽(Intel Iris Xe)으로는 모델 학습이 불가능해 코랩을 활용함. 하지만 T4 GPU는 bfloat16을 지원하지 않아 Engine core initialization failed 에러가 발생.

해결: vLLM LLM 객체 생성 시 dtype="float16"으로 정밀도를 강제 고정하여 해결.

문제 2: vLLM의 무자비한 메모리 점유 (OOM)
기본 설정으로 모델을 띄우려니 GPU 메모리가 부족해 엔진이 자꾸 터짐.

해결: gpu_memory_utilization 값을 0.75에서 0.4까지 극한으로 낮추고, max_model_len을 1024로 줄여 메모리 다이어트에 성공함. (enforce_eager=True 옵션도 켜서 메모리 낭비 방지)

3. 배운 점

vLLM은 실행 즉시 GPU 메모리를 미리 찜(allocation)해버리기 때문에, 제한된 자원(T4 16GB)에서는 옵션 타협이 필수적이라는 것을 몸소 깨달음.

📅 [2/26] Git/GitHub 트러블슈팅: 과거의 망령(APK) 퇴치 작전
1. 오늘 한 일

기존 프로젝트 코드를 GitHub develop 브랜치로 통째로 마이그레이션 (git push)

GitHub 100MB 이상 대용량 파일 업로드 제한(Large files detected) 이슈 해결

2. 겪었던 문제 & 해결 과정 (Troubleshooting)

이슈 발생: 로컬에서 git push github develop을 실행했으나, 과거에 빌드했던 app-debug.apk 파일(144MB) 때문에 GitHub 문지기에게 컷 당함.

시도와 실패: 로컬 폴더에서 APK 파일을 지우고 다시 push 했으나 실패. Git은 '사진첩' 같아서, 현재 지웠더라도 과거 커밋 내역(기억) 속에 파일이 남아있으면 push가 안 된다는 것을 알게 됨.

해결 (기억 조작): git filter-branch --force --index-filter "git rm --cached --ignore-unmatch 경로/app-debug.apk" --prune-empty --tag-name-filter cat -- --all
명령어를 사용해 과거의 모든 커밋 내역에서 해당 APK 파일의 흔적만 싹 도려냄. 이후 git push -f (강제 푸시)를 통해 깃허브로 이사 완벽하게 성공!

3. 배운 점

Git의 commit은 단순한 저장이 아니라 '스냅샷'의 연속이라는 개념을 완벽하게 이해함. 빌드 결과물(APK)이나 민감한 파일은 애초에 .gitignore에 꼭 넣어야 함을 뼈저리게 느낌.
