# MVC, MVP, MVVM 아키텍처 패턴 비교 분석

애플리케이션을 개발할 때 사용자 인터페이스(UI)와 비즈니스 로직을 분리하는 것은 코드의 재사용성, 테스트 용이성, 그리고 유지보수성을 높이는 데 매우 중요합니다. 이를 위해 고안된 대표적인 디자인 패턴인 MVC, MVP, MVVM의 개념과 차이점을 알아봅니다.

---

## 1. MVC (Model - View - Controller)
가장 기본적이고 널리 사용되는 디자인 패턴입니다. 사용자의 동작이 Controller로 들어옵니다.

### 구성 요소
* **Model (모델)**: 애플리케이션의 데이터와 비즈니스 로직을 담당합니다.
* **View (뷰)**: 사용자에게 보여지는 UI 화면입니다. 모델의 데이터를 바탕으로 화면을 그립니다.
* **Controller (컨트롤러)**: 사용자의 입력(Action)을 받고 처리합니다. 모델의 데이터를 업데이트하거나 뷰를 제어합니다.

### 동작 흐름
1. 사용자의 액션이 Controller에 들어옵니다.
2. Controller는 사용자의 액션을 확인하고, Model을 업데이트합니다.
3. Controller가 View를 선택하여 화면을 업데이트하거나, Model이 업데이트되면 View가 이를 감지하여 화면을 갱신합니다.

### 장단점
* **장점**: 가장 단순하며 널리 보편적으로 사용되어 개발자들이 쉽게 적응할 수 있습니다.
* **단점**: View와 Model 사이의 의존성이 높습니다. 애플리케이션이 복잡해질수록 Controller가 비대해지는 Massive View Controller 문제가 발생하기 쉽습니다.

---

## 2. MVP (Model - View - Presenter)
MVC의 단점인 View와 Model의 의존성을 해결하기 위해 등장한 패턴입니다. 사용자의 동작이 View로 들어옵니다.

### 구성 요소
* **Model**: 데이터와 비즈니스 로직 담당. (MVC와 동일)
* **View**: 사용자에게 보여지는 화면 담당. (MVC와 동일)
* **Presenter (프리젠터)**: View에서 요청한 정보를 Model로부터 가공하여 View에 전달해 주는 중간 매개체 역할을 합니다.

### 동작 흐름
1. 사용자의 액션이 View를 통해 들어옵니다.
2. View는 데이터를 Presenter에 요청합니다.
3. Presenter는 Model에게 데이터를 요청하고, Model은 Presenter에게 데이터를 응답합니다.
4. Presenter는 응답받은 데이터를 가공하여 View로 보냅니다.

### 장단점
* **장점**: Presenter를 통해 View와 Model을 완전히 분리하여 의존성을 없앴습니다. UI를 제외한 로직의 유닛 테스트가 쉬워집니다.
* **단점**: View와 Presenter가 1:1로 강하게 결합되어 있어, 애플리케이션이 복잡해질수록 View와 Presenter 간의 의존성이 커지는 새로운 문제가 발생합니다.

---

## 3. MVVM (Model - View - ViewModel)
MVP의 View와 Presenter 간의 의존성 문제까지 해결하기 위해 고안된 패턴입니다. WPF, Android 등 최신 프레임워크에서 많이 권장됩니다. **데이터 바인딩(Data Binding)**이 핵심입니다.

### 구성 요소
* **Model**: 데이터와 비즈니스 로직 담당. (MVC, MVP와 동일)
* **View**: 사용자에게 보여지는 화면 담당. ViewModel의 상태를 관찰(Observe)하여 화면을 갱신합니다.
* **ViewModel (뷰모델)**: View를 표현하기 위해 만든 View를 위한 Model입니다. View에 바인딩될 데이터와 명령(Command)을 가지고 있습니다.

### 동작 흐름
1. 사용자의 액션이 View를 통해 들어옵니다.
2. View에 액션이 들어오면, Command 패턴으로 ViewModel에 액션을 전달합니다.
3. ViewModel은 Model에게 데이터를 요청하고, Model은 ViewModel에게 응답합니다.
4. ViewModel은 응답받은 데이터를 가공하여 저장합니다.
5. View는 ViewModel과의 **데이터 바인딩(Data Binding)**을 통해 자동으로 상태 변화를 감지하고 화면을 갱신합니다.

### 장단점
* **장점**: View와 Model, View와 ViewModel 간의 의존성이 모두 없습니다 (View와 ViewModel은 n:1 관계 가능). 독립성이 매우 높아 모듈화와 테스트에 가장 적합합니다.
* **단점**: 데이터 바인딩 등 초기 설정 및 설계가 복잡하며, 러닝 커브(Learning Curve)가 높은 편입니다.

---

## 💡 한눈에 보는 요약 비교표

| 비교 항목 | MVC | MVP | MVVM |
| :--- | :--- | :--- | :--- |
| **사용자 입력 위치** | Controller | View | View |
| **핵심 매개체** | Controller | Presenter | ViewModel |
| **View - Model 의존성** | 높음 | 없음 (Presenter가 분리) | 없음 (ViewModel이 분리) |
| **View - 매개체 관계** | 1 : n | 1 : 1 (강한 결합) | n : 1 (의존성 낮음) |
| **핵심 특징** | 단순하고 보편적 | 인터페이스를 통한 통신 | 데이터 바인딩, Observer 패턴 |
| **가장 큰 단점** | Controller 비대화 | Presenter 비대화, 1:1 결합 | 설계의 복잡성, 높은 러닝 커브 |
