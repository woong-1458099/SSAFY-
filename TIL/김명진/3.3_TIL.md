# 📝 [TIL] 2026.03.03: 저장 프로시저(Stored Procedure)의 깊은 이해 및 View, Trigger와의 비교

## 1. 저장 프로시저(Stored Procedure, SP)란?
저장 프로시저(SP)는 일련의 복잡한 SQL 쿼리문들을 마치 하나의 함수나 메서드처럼 RDBMS 내부에 묶어서 저장해 두고 사용하는 쿼리의 집합입니다. 

### 💡 SP의 핵심 동작 원리 (캐싱과 실행 계획)
* **최초 실행 시**: SQL 서버는 해당 프로시저의 구문을 분석하고, 최적화(Optimization) 및 컴파일(Compile) 단계를 거칩니다. 이 과정을 통해 생성된 **'실행 계획(Execution Plan)'이 데이터베이스의 메모리(캐시)에 저장**됩니다.
* **이후 실행 시**: 이미 캐싱된 실행 계획을 그대로 가져와서 사용합니다. 즉, 매번 길고 복잡한 SQL 구문을 다시 해석하고 최적화하는 과정이 생략되기 때문에 **실행 속도가 비약적으로 향상**됩니다.
* 단, 쿼리 해석 과정을 건너뛸 뿐, 실제 데이터를 디스크에서 읽고 쓰는 물리적인 입출력(Data I/O) 작업은 쿼리의 목적에 따라 반드시 일어납니다.

---

## 2. 저장 프로시저 vs 일반 SQL문 비교
어플리케이션(서버)과 DB 간의 통신 방식을 비교하면 그 차이가 명확해집니다.

### ❌ 일반적인 SQL문 실행 방식
* **과정**: 서버 어플리케이션에서 직접 복잡한 SQL 쿼리를 DB로 전송합니다.
* **비효율성**: DB는 조건에 맞는 대량의 원천 데이터(Raw Data)를 모두 찾아 서버로 반환하고, 서버는 CPU와 메모리를 사용해 이 데이터를 다시 가공해야 합니다. 데이터 양이 많을수록 **네트워크 트래픽 부하**가 극심해집니다.
* **캐싱의 한계**: 쿼리의 검색 구문이나 띄어쓰기가 한 글자라도 달라지면 DB는 이를 완전히 새로운 쿼리로 인식하여 매번 무거운 최적화와 컴파일 과정을 반복해야 합니다.

### ⭕ 저장 프로시저(SP) 실행 방식
* **과정**: 어플리케이션 서버는 오직 `EXEC 프로시저명(매개변수)` 형태의 짧은 호출 명령과 인자값만 DB로 전송합니다.
* **효율성**: 실제 복잡한 조인(Join), 조건문(If/Else), 반복 연산 등은 모두 DB 내부에서 처리되며, 서버로는 **최종적으로 가공이 완료된 결과값만 반환**됩니다. 
* **캐싱 활용**: 매개변수(인자) 값만 달라질 뿐 프로시저 자체는 동일하므로, 1회 호출 시 만들어둔 메모리 캐시를 계속 재사용하여 서버 리소스를 아낄 수 있습니다.

## 3. 저장 프로시저를 도입했을 때의 핵심 장점

1. **DB 및 어플리케이션 성능 향상**
   * 앞서 언급한 실행 계획 캐싱 덕분에 쿼리 응답 속도가 빠릅니다. 또한, 데이터 연산을 DB 서버에서 직접 처리하므로 어플리케이션 서버의 부하를 덜어줄 수 있습니다.
2. **네트워크 부하의 극적인 감소**
   * 수백 줄에 달하는 복잡한 쿼리문을 매번 네트워크를 통해 전송할 필요 없이, 단 한 줄의 프로시저 호출문만 보내면 되므로 네트워크 대역폭 소모가 크게 줄어듭니다.
3. **유지보수의 편의성과 로직의 독립성**
   * 데이터베이스 로직과 어플리케이션 서버 로직을 완벽하게 분리할 수 있습니다. 쿼리 로직을 수정해야 할 때, 서버 코드를 고치고 재배포(재가동)할 필요 없이 DB에 접속해 SP의 내부 로직만 `ALTER` 해주면 즉시 반영됩니다.
4. **강력한 보안 및 권한 통제**
   * 사용자(혹은 어플리케이션)에게 테이블에 직접 접근(SELECT, UPDATE 등)할 수 있는 권한을 주지 않고, **특정 저장 프로시저만 실행할 수 있는 권한(EXECUTE)만 부여**할 수 있습니다. 이를 통해 민감한 원본 데이터 테이블을 안전하게 숨길 수 있으며, SQL 인젝션(SQL Injection) 공격을 예방하는 데에도 탁월합니다.
5. **실무에서의 높은 활용도**
   * 데이터 정합성이 중요하고 복잡한 비즈니스 로직이 얽혀 있는 금융권, 대기업 ERP, 게임 서버 등에서 필수적으로 다루는 기술입니다.

## 4. [심화] Stored Procedure vs View vs Trigger 전격 비교

데이터베이스에서 목적에 따라 헷갈리기 쉬운 세 가지 객체를 비교해 보았습니다.

| 구분 | 저장 프로시저 (Stored Procedure) | 뷰 (View) | 트리거 (Trigger) |
| :--- | :--- | :--- | :--- |
| **정의** | 일련의 쿼리를 함수나 메서드처럼 묶어둔 **프로그램** | 하나 이상의 테이블을 조인하여 만든 **가상의 테이블** (Virtual Table) | 테이블에 특정 이벤트 발생 시 **자동으로 실행되는 특수한 프로시저** |
| **호출 및 실행 방식** | 사용자가 `EXEC` 또는 `CALL` 명령어로 **직접(명시적) 호출** | 일반 물리 테이블처럼 `SELECT` 문을 통해 **조회** | 사용자가 직접 호출할 수 없으며, 이벤트(`INSERT`, `UPDATE`, `DELETE`) 발생 시 **자동(암묵적)으로 실행** |
| **파라미터 (인자) 전달** | 가능 (IN, OUT 매개변수를 통해 값을 받고 결과 반환 가능) | 불가능 (단순히 쿼리를 저장해둔 껍데기 역할) | 불가능 (발생한 이벤트의 맥락에 종속적으로 동작) |
| **주요 사용 목적** | - 복잡한 비즈니스 로직의 캡슐화<br>- 네트워크 트래픽 감소 및 성능 최적화 | - 복잡하고 긴 JOIN 쿼리의 단순화<br>- 특정 사용자에게 필요한 컬럼만 노출하는 **보안(데이터 은닉)** | - 데이터 무결성 유지 (ex. 참조 데이터 연쇄 수정)<br>- 데이터 변경 이력(Log/History)의 **자동 기록 및 감사** |
| **특징 및 주의점** | 디버깅이 어플리케이션 코드보다 상대적으로 어려울 수 있음 | 실제 데이터를 저장하지 않으며 쿼리문만 저장함 (단, Materialized View 제외) | 남발할 경우 어떤 타이밍에 변경이 일어났는지 추적하기 힘들어져 **유지보수 헬(Hell)**을 유발할 수 있음 |

### 📌 요약하자면?
* **능동적인 작업 지시**가 필요하고 복잡한 로직을 수행해야 한다면 ➡️ **저장 프로시저 (SP)**
* 반복적이고 복잡한 조인 결과를 **마치 하나의 테이블처럼 쉽게 조회**하고 싶다면 ➡️ **뷰 (View)**
* "A 테이블에 데이터가 추가되면, B 테이블에 로그를 남겨라" 처럼 **자동화된 연동 작업**이 필요하다면 ➡️ **트리거 (Trigger)**
