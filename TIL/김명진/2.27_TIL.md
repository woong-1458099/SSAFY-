# TIL: HTTP 통신과 Socket 통신의 차이점 및 게임 서버 활용 예시

**학습 일자**: 2026.02.27

## 1. HTTP 통신의 특징 (단방향, Stateless)
* **단방향 통신**: 클라이언트(Client)의 요청(Request)이 있을 때만 서버(Server)가 응답(Response)을 반환합니다. 주로 텍스트 기반의 데이터를 주고받습니다.
* **Stateless Protocol**: 서버가 클라이언트의 상태를 기억하지 않습니다. 데이터를 요청할 때마다 연결(Connect)과 종료(Close)를 반복합니다.
* **데이터 관리**: 서버 자체 메모리에 유저 데이터를 저장하지 않고, 반드시 DB(또는 Redis 등 메모리 DB)를 거쳐 데이터를 주고받아야 합니다.

## 2. Socket (소켓) 통신의 특징 (양방향, Stateful)
* **양방향 통신**: 네트워크상의 두 프로그램 간 접속점(소켓)을 통해 패킷 단위로 데이터를 주고받습니다. 특정 포트를 열어두고 실시간 양방향 통신을 진행합니다.
* **Stateful Protocol**: 양측에서 임의로 연결을 끊지 않는 한 지속적으로 연결 상태를 유지하며, 서버가 클라이언트의 상태를 기억합니다.
* **데이터 관리**: DB를 거칠 수도 있지만, 서버 프로그램 자체 메모리에 유저 데이터를 임시로 생성하고 실시간으로 갱신 및 관리할 수 있습니다.

## 3. 소켓 통신의 실제 활용 예시: 게임 PvP 서버 구현
소켓 통신은 서버 메모리에 데이터를 올려두고 실시간으로 패킷을 주고받는 데 특화되어 있어 실시간 대전 게임(PvP)에 필수적입니다. 소켓 통신을 활용한 서버의 상세 동작 흐름은 다음과 같습니다.

**① 메모리 자료 구조 세팅**
* 클라이언트와 서버는 완벽히 동일한 구조의 `PlayerInfo` 데이터(Index, HP, Power, State 등)를 가집니다.
* 서버 메모리상에는 유저들을 연결해주기 위한 **'대기 큐'**와 게임 방 관리를 위한 **'맵(Map)'**이 상시 구동되고 있습니다.

**② 실시간 매칭 및 방 입장**
* 두 명의 플레이어(Client 1, Client 2)가 입장을 요청하면, 서버가 이들을 매칭하여 'Room 1'을 생성합니다.
* 서버는 메모리에 두 플레이어의 초기 상태를 세팅하고, 양쪽 클라이언트에 서로의 정보를 패킷으로 전송하여 양측 화면의 데이터를 동기화합니다.

**③ 실시간 공격 및 데미지 동기화**
* Client 1이 공격 버튼을 눌러 패킷을 서버로 전송합니다.
* DB를 거칠 필요 없이, 서버는 자체 메모리상에서 데미지를 계산하여 Client 2의 HP를 100에서 50으로 즉시 깎습니다.
* 서버는 이 변경된 HP 정보를 양쪽 클라이언트에 동시에 쏴주어(전송하여), 두 유저의 화면에 딜레이 없이 체력 감소가 반영되도록 합니다.

**④ 무적 스킬 사용과 상태(State) 실시간 제어**
* 불리해진 Client 2가 '무적 스킬' 사용 패킷을 서버로 보냅니다.
* 서버는 즉시 Client 2의 상태(State) 변수를 '보통'에서 '무적'으로 변경하고, 서버 내부에서 5초 타이머를 가동합니다.
* 5초 후 타이머가 종료되면 서버는 다시 상태를 '보통'으로 되돌리고, 이를 양쪽 클라이언트에 알려 무적 상태가 끝났음을 완벽하게 동기화합니다.

## 💡 결론 및 요약
웹 브라우저의 단방향 요청(HTTP)과 달리, **소켓 통신은 서버가 직접 연결 상태와 유저 데이터를 메모리에 들고 있으면서, 실시간으로 변하는 급박한 상황(공격, 스킬, 체력 변화 등)을 양쪽 클라이언트에 즉각적이고 지속적으로 동기화해 주는 강력한 역할**을 수행합니다.
