## 오늘 생긴 문제
- 공용 nginx를 분리한 뒤 처음에는 `prod-api-blue`, `stg-api-blue`를 찾지 못해 nginx가 재시작 루프에 들어갔다.
- 원인은 app 컨테이너 alias가 실제로 반영되지 않은 상태였기 때문이다.
- backend Jenkinsfile의 `Verify And Rollback` 단계에서 ssh 문자열과 로컬 shell 변수가 섞여서 `docker: not found`, `rollback_color: unbound variable` 에러가 났다.
- backend path filter는 `currentBuild.changeSets`에만 의존하고 있어서 master 쪽 merge 이벤트에서 변경 파일을 못 잡는 경우가 있었다.
- prod DB 접근 시 `Role "prod_app" does not exist` 에러가 발생했다.
- 이후 확인해보니 이번 구조는 STG / PROD가 DB를 분리하지 않고 `stg_app` DB를 함께 사용하는 전제였기 때문에, prod 계정 분리 자체가 오히려 혼선을 만들었다.
- IntelliJ DB 연결은 Docker 내부 네트워크와 호스트 프로세스의 차이 때문에 바로 붙지 않았고, dev container를 점검용으로 쓰는 방향을 검토했다.
- JetBrains remote / dev container 환경에서는 IDE 내부 예외가 발생해 DB GUI보다는 터미널 기반 확인이 더 현실적이라는 점도 확인했다.

## 해결한 방법
- nginx는 `compose.app.yml`에서 완전히 제거하고 `compose.nginx.yml`로 독립시켰다.
- app compose에는 alias를 추가해 공용 nginx가 STG / PROD를 명확히 구분하도록 했다.
- upstream 파일을 `active-stg.conf`, `active-prod.conf`로 나눴다.
- backend Jenkins verify/rollback은 `ssh heredoc` 방식으로 바꿔 로컬 shell 평가 문제를 제거했다.
- Jenkinsfile 이름과 job 이름을 정리하고, path filter에도 각자 자기 Jenkinsfile 경로를 넣었다.
- branch guard를 추가해 develop / master 대상이 아닌 MR에서는 `NOT_BUILT`로 빠지도록 정리했다.
- DB는 분리하지 않는 전제로 다시 맞추고, `.env.prod`도 `stg_app` DB / 계정을 공유하는 기준으로 보는 방향을 확인했다.

## 배운 점
- 같은 EC2에서 STG / PROD를 공존시키더라도 nginx, upstream, frontend path, Jenkins job, webhook을 분리하면 운영 경계를 꽤 명확하게 만들 수 있다.
- Docker network alias는 공용 ingress 구조에서 환경별 backend를 식별하는 데 매우 중요하다.
- Jenkins에서 원격 명령은 `ssh "..."`에 로컬 변수와 명령 치환을 섞지 말고, 가능하면 `ssh ... <<'EOF'` 형태로 통일하는 것이 안전하다.
- `currentBuild.changeSets`는 GitLab webhook / merge 상황에 따라 비어 있을 수 있어서, path filter는 `git diff` 기반 fallback이 더 실용적이다.
- DB를 하나로 공유할 거면 계정/권한 정책도 같이 단순화해야 한다. DB는 공유하면서 계정만 분리하면 Flyway 같은 곳에서 권한 충돌이 쉽게 난다.
- Docker 내부 서비스명은 컨테이너끼리는 잘 되지만, 호스트 프로세스나 IntelliJ 같은 도구에서는 그대로 쓸 수 없다.
- Dev container는 DB 점검용으로는 유용할 수 있지만, IDE GUI 전체를 기대하기보다는 네트워크 점검용 터미널 환경으로 보는 게 더 현실적이다.
